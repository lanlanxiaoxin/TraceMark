# TraceMark 产品评审与提升方案

> 评审视角：15 年资深产品经理 · 独立产品观察 · 不为现有方案背书
> 评审时间：2026-05-21
> 评审方法：idea-refine（重述 → 扩散 → 收敛 → 压力测试 → MVP 收口）
> 评审范围：产品定位、主流程、用户价值、隐私设计、工程基线、留存模型

---

## 0. 一页结论（TL;DR）

| 维度 | 现状判断 | 建议动作 |
|------|---------|---------|
| **产品定位** | "个人工作资产账本"概念锐利但承诺过宽 | **收敛为"绩效叙事 + 今日盖章"双锚点**，B/C 方向延后 |
| **主流程** | 候选 → 确认 → 沉淀，理论闭环但确认负担高 | "今日"页重构为 **5 分钟睡前仪式**，从堆积转盖章 |
| **隐私设计** | L0–L3 + 上传预览，原则一流 | MVP **暂停 L3**，把 L1/L2 简化为"基础/增强"两档 |
| **用户战略** | 程序员 + PM 双线并列 | **MVP 只服务程序员**，PM 作为兼容场景 |
| **工程基线** | typecheck 已升级到 `tsc -b` | 补 e2e 烟雾测试 + 核心指标埋点 dashboard |
| **核心风险** | 留存机制空缺 + 价值兑现滞后 | 加"今天就有用"瞬时价值锚（周五叙事） |

**最重要的一句话：**

> TraceMark 当前最大的风险不是技术、不是隐私、不是采集质量——而是**用户为什么明天还要打开它**。先解决"明天"和"今晚"这两个问题，再谈"三个月后的知识库"。

## 1. 问题陈述（HMW）

> **How might we** 让一个独立开发的"个人工作资产账本"，在用户没有外部绩效压力的日常里，依然让他每天主动打开、并相信这里沉淀的东西未来真的值钱？

这是 TraceMark 真正未解决的产品命题。技术、隐私边界、采集机制都已经在路上；但**"留存"和"价值兑现"两个产品命脉问题，当前的设计没有给出答案**。

---

## 2. 现状盘点（褒贬并陈）

### 2.1 现状优势

1. **CLAUDE.md 的产品自律性极强** —— 红线清晰、不做清单详尽、隐私边界明确。这在独立产品里非常罕见。
2. **核心闭环已落地** —— 采集 → 候选生成 → 用户确认 → 资产库 → 复盘，端到端跑通。
3. **三类卡片（成果/过程/证据）分类清晰** —— 比单一"待办"或"日志"模型更有产品味道。
4. **分级授权（L0–L3）框架成熟** —— 在独立桌面工具里属于第一梯队的隐私设计。
5. **工程基线在好转** —— `tsc -b --noEmit` 已经替代了原来不可靠的 typecheck 脚本。

### 2.2 现状隐患

| # | 隐患 | 严重度 |
|---|------|--------|
| H-1 | "确认 5–8 张卡/天"的体验，对个人用户负担过重 | 🔴 P0 |
| H-2 | 缺少"今天就有用"的瞬时价值，资产沉淀是滞后回报 | 🔴 P0 |
| H-3 | 没有任何机制保证 confirmed 资产会被回看/复用 | 🔴 P0 |
| H-4 | 概念太多（项目空间/别名/隐私别名/三类卡片/四档授权/上传预览…）认知负担过高 | 🟠 P1 |
| H-5 | 程序员 + PM 双用户并列，证据源差异巨大，难以同时做深 | 🟠 P1 |
| H-6 | 隐私强度 ↔ AI 候选质量存在内在张力，MVP 阶段会打架 | 🟡 P2 |
| H-7 | MoSCoW 的 Must 列表过宽（10+ 项），没有真正的产品 wedge | 🟡 P2 |
| H-8 | 留存机制为零（无社交、无通知唤醒、无 Agent 主动推送） | 🟡 P2 |

## 3. 产品方向扩散（5 种身份）

下面 5 个方向**对应完全不同的产品身份**，彼此互斥，必须主动选一。

| # | 方向身份 | 核心动作 | 主用户 | 数据飞轮 | 与 TraceMark 当前的距离 |
|---|---------|---------|--------|---------|------------------------|
| **A** | **绩效叙事代笔人** | 周五一键生成"我这周干了什么"，可发老板/写绩效 | 大厂程序员 & PM | 一周一次主动打开 | 近，需要把周复盘做到极致 |
| **B** | **个人知识库底座** | 把碎片沉淀成可检索资产，3 个月后能复用 | 资深从业者、咨询、独立创作者 | 资产越多越值钱 | 当前主线声称走 B，但**未解决复用** |
| **C** | **项目复盘教练** | 项目结束自动重建 timeline + 决策点 + 反思 | 同时跑 3+ 项目的 PM/实施 | 项目级使用，非日活 | 已有项目阶段复盘骨架 |
| **D** | **AI 副驾的私域记忆** | 桌面 Agent 的长期记忆，喂给 LLM 做上下文 | AI 重度用户、Prompt 工程师 | 记忆越完整 AI 越准 | 远，但 6–12 个月的下一波 |
| **E** | **"今天我做了什么"焦虑解药** | 每晚 5 分钟，盖章/补充，今天就此封存 | 远程办公、自由职业、独立创作者 | 心理-情感型留存 | 远，但**最能解决留存** | 

### 关键观察

- 当前 CLAUDE.md 像在 **A + B + C 全要**，没有把 D 和 E 当作严肃方向。
- **D 和 E 是当前被严重低估的方向**：D 紧贴 AI 浪潮，E 是真正的"个人工具心智"。
- B 方向的承诺最难兑现（资产沉淀的复用接口在 MVP 阶段根本做不深）。

## 4. 压力测试（关键假设）

| # | 假设 | 当前证据 | 风险 | 验证方式 |
|---|------|---------|------|---------|
| **A1** | 用户愿意每天确认 5–8 张卡片 | 无 | 🔴 极高 | 5 个真实用户跑 2 周，看确认率与停用率 |
| **A2** | confirmed 资产会在未来被回看/复用 | 无 | 🔴 极高 | 埋点：30 天内资产被打开/搜索次数 |
| **A3** | 自动生成的候选卡片质量足够（≥70% 正确） | 已实现未规模验证 | 🟠 高 | 让 5 个用户人工标注 100 张候选 |
| **A4** | 程序员和 PM 能共用一套资产模型 | 设计上"是"，未验证 | 🟡 中 | 分群对比留存与确认率 |
| **A5** | 隐私分级是用户选择留下的关键卖点 | 偏向产品自我感动 | 🟡 中 | 用户访谈：是否因隐私功能而留 |
| **A6** | 周复盘/阶段复盘是真实主动需求 | 中等（绩效场景成立） | 🟡 中 | 看复盘功能 14 日内使用率 |

> **A1 + A2 是产品命脉。若都不成立，整个"工作资产账本"叙事需重写。**

## 5. 推荐方向

### 5.1 主线收敛

> **现阶段把主线收敛为 A（绩效叙事代笔人）+ E（今日盖章仪式）的组合**
> **主动放弃在 MVP 阶段把 B（知识库）做深**
> **PM 用户作为兼容场景，主线只服务程序员**

理由：

1. **A 解决"明天为什么打开"** —— 周五的绩效焦虑是真实的、周期性的、有付费意愿的。
2. **E 解决"今天为什么打开"** —— 把"今日"页改造为 5 分钟睡前仪式，而不是待办堆积箱。
3. **B 在 MVP 做不深** —— 知识库需要资产体量、复用界面、检索深度，3 个月做不出来。
4. **D 是 6–12 个月的下一波** —— 等 MCP/Agent 记忆生态更成熟。
5. **聚焦程序员** —— Git/IDE 信号强，证据结构化，第一波最容易做出"哇塞感"。

### 5.2 "今日"页体验重构（关键改造）

**当前（堆积式）：**

```
今日 5 张候选卡（可能 0 张）
  → 用户一张张确认/忽略/拆分/合并
  → 列表越长越焦虑
  → 不看就堆积
```

**建议（仪式式）：**

```
[ 今晚 5 分钟，给今天盖个章 ]

🌗 你今天的主线似乎是：[ 项目 A · 调试登录态 BUG ]
   ▢ 是的，确认  ▢ 不是，换一个  ▢ 跳过

⭐ 今天最值得记下的一件事？
   [ 一句话输入框，可以为空 ]

📌 系统帮你存了 3 张候选证据（Git、文档、浏览器）
   [ 展开 ]  [ 一键全部归档进项目 ]  [ 不要 ]

[ 完成 → 自动生成今日盖章卡 · 显示连续 N 天 ]
```

**关键变化：**

| 维度 | 当前 | 建议 |
|------|------|------|
| 主动权 | 系统驱动确认 | 用户先表态主线 |
| 默认完成 | 不可达（卡多就累） | 可达（默认 5 分钟） |
| 留存钩子 | 无 | 连续盖章天数 |
| 失败兜底 | 卡片堆积焦虑 | 跳过即关闭，明天再来 |
| 价值锚 | 滞后（等周复盘） | 即时（仪式感本身就是价值） |

### 5.3 周复盘升级为"绩效叙事"

**当前：**
- 周复盘是基于 confirmed 资产生成的总结文本
- 偏报告口吻、偏内部归纳

**建议：**
- 改名（产品内）为"**本周战报**"或"**周五叙事**"
- 模板分两版：**"给老板看"** 与 **"给自己看"**
- 强化主动权：周五下午 5 点桌面通知，一键打开预览，三段式（成果/突破/卡点）
- 关键指标："周五打开率"必须 ≥ 50% 才算 PMF 信号

### 5.4 项目空间设置简化

**当前需要填的：**
- 项目名 + 别名 + 隐私别名 + Git 路径 + 浏览器关键词 + 文档关键词 + 文档目录 + 会议关键词 + 聊天关键词 + 名字别名 + 角色模板

**建议三步走（首次设置 ≤ 30 秒）：**

```
Step 1: 项目叫什么？    [ TraceMark ]
Step 2: 它的代码在哪？  [ 选择文件夹 ] (可跳过)
Step 3: 还有别的关键词吗？(高级)
```

其余字段统统折叠到二级"高级设置"，并提供智能默认（用项目名做关键词）。

## 6. MVP 范围（建议未来 8 周）

### 6.1 Do（明确要做）

| # | 项 | 验收标准 |
|---|---|---------|
| D1 | "今日盖章"流，替代当前 TodayInbox 主流程 | 5 分钟内可完成，可跳过 |
| D2 | "周五战报"AI 一键生成 + 桌面通知 | 周五打开率埋点 ≥ 50% |
| D3 | 项目空间设置简化为三步 | 首次设置中位数 ≤ 30 秒 |
| D4 | Git enrichment 打磨 | commit/PR/issue 解析准确率 ≥ 85% |
| D5 | L1 + L2 简化为"基础/增强"两档 | 用户能用一句话解释两档区别 |
| D6 | 核心指标埋点 dashboard（本地） | 至少 6 个核心漏斗指标可见 |
| D7 | e2e 烟雾测试（采集→候选→确认→复盘） | CI 中常态运行 |
| D8 | "连续盖章天数"留存钩子 | 中断时显示提示，但不强行打扰 |

### 6.2 Don't（明确不做 —— 这是最贵的产出）

| # | 暂不做 | 原因 |
|---|--------|------|
| ❌ N1 | 双用户深度做（PM 的 Figma/会议集成） | 先把程序员做透 |
| ❌ N2 | 项目阶段复盘的深度打磨 | 高级功能，等周复盘 PMF 验证 |
| ❌ N3 | 本地轻量模型集成 | 架构预留即可，MVP 不依赖 |
| ❌ N4 | L3 项目目录文档扫描 | 隐私风险高、MVP 收益低 |
| ❌ N5 | macOS / Linux 第一方支持 | Windows 先打透，多平台保留抽象不实现 |
| ❌ N6 | 资产搜索/标签/复用界面（B 方向） | 资产体量不够前不做，避免空心化 |
| ❌ N7 | 复杂的合并/拆分流程深度优化 | 80% 用户不会用，保留轻量版即可 |
| ❌ N8 | 上传预览 dialog 的复杂结构 | 简化为一行说明 + 折叠详情 |
| ❌ N9 | 团队/分享/同步功能 | 红线 |
| ❌ N10 | 通过 OCR / 截图增强采集 | 红线 |

### 6.3 用 SMART 重写"成功标准"

> 8 周后，TraceMark 在 5 个真实程序员用户上做到：
> - **第 7 日留存 ≥ 60%**
> - **第 30 日留存 ≥ 35%**
> - **周五战报打开率 ≥ 50%**
> - **每日今日盖章完成率 ≥ 40%（默认 5 分钟内）**
> - **候选卡片确认率 ≥ 50%，忽略率 ≤ 30%**
> - **30 天内 confirmed 资产被搜索/打开次数 ≥ 1（验证 A2）**

任何一项不达标 → 触发产品方向二次评审，不要硬撑加功能。

## 7. 工程提升建议（与产品并行）

### 7.1 立即做

1. **核心指标埋点** —— 当前 `recordMetric('page_visit')` 太弱，至少补：
   - `today_seal_started` / `today_seal_completed` / `today_seal_skipped`
   - `asset_confirmed` / `asset_ignored` / `asset_split` / `asset_merged`
   - `weekly_retro_opened` / `weekly_retro_exported`
   - `asset_library_search` / `asset_library_reopen` (验证 A2 命脉)
2. **本地指标 dashboard** —— 哪怕只是一个 SQL 视图 + 简单页面，让你自己每周能看一眼漏斗。
3. **e2e 烟雾测试** —— `smoke-pro50.mjs` 升级或并行加一个真实跑通"采集 → 候选 → 确认 → 复盘"的脚本。

### 7.2 中期做

4. **错误监控** —— 即使是个人工具，Sentry / 本地 crash log 仍有必要。
5. **自动化打包** —— GitHub Actions 触发 Windows 安装包构建，方便分发给试用用户。
6. **AI Gateway 抽象稳定** —— 让用户配置 OpenAI 兼容接口的体验做到 60 秒内可用。

### 7.3 暂不做

7. ❌ 引入状态管理库（Redux/Zustand）—— React 内置够用。
8. ❌ 重构 Electron 主进程为微内核架构 —— 过度设计。
9. ❌ 引入 ORM —— better-sqlite3 直接写 SQL 更可控。

## 8. 风险与缓解

| 风险 | 触发条件 | 缓解措施 |
|------|---------|---------|
| **A1/A2 假设破产** | 8 周内留存指标不达标 | 准备"今日盖章"作为方向 E 的轻量化退路 |
| **AI 候选质量低于预期** | 用户确认率 < 30% | 优先打磨 Git enrichment（程序员主线，信号强） |
| **隐私强度反噬体验** | 用户抱怨候选卡片"没有信息" | 引导用户主动开启 L2 增强；首屏 onboarding 中说清楚收益 |
| **作者精力不足** | 独立开发者一人多角 | 严格执行 Not Doing 清单；任何新功能默认拒绝 |
| **AI 浪潮冲击产品形态** | LLM Agent 出现"通用桌面记忆" | 6 个月评审一次，预留方向 D 切换路径 |

## 11. 附录：用 idea-refine 框架做评审的好处

这份评审之所以采用 idea-refine 方法（重述 HMW → 扩散 5 种身份 → 压力测试假设 → 收敛 + Not Doing），原因有三：

1. **强制暴露假设** —— 大多数独立产品死于"我以为用户会怎样"。
2. **强制写 Not Doing 清单** —— 比 To Do 清单更有价值，明确放弃才能聚焦。
3. **强制定义成功标准** —— 没有可证伪的成功标准，6 个月后无法判断是否该转方向。
